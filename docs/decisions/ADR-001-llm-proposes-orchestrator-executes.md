# ADR-001: LLM은 제안만, 실행은 오케스트레이터만

- 상태: 채택 (2026-07-03)

## 결정

LLM 출력이 직접 상태를 바꾸는 코드 경로를 만들지 않는다. LLM은
`{"intent": ..., "slots": {...}, "confidence_signals": [...]}` 형태의 **제안**을 반환하고,
오케스트레이터가 pydantic 스키마 검증 → 슬롯 스키마 대조 → 상태머신 전이 테이블 검증을
통과시킨 것만 적용한다.

## 근거

인쇄는 비가역이다. 잘못된 슬롯 확정 1건 = 실물 파손 = 실비 손실 + 신뢰 손실.
LLM의 언어 능력은 쓰되, 결정 권한을 주지 않으면 오류가 "이상한 질문" 수준에서 멈추고
"잘못된 주문" 까지 가지 않는다.

## 구현 규칙

1. LLM 호출부(`core/llm/`)는 오케스트레이터·상태머신·DB를 임포트할 수 없다 (의존 방향 단방향).
2. LLM 출력은 반드시 `core/llm/parsing.py`의 pydantic 모델을 통과한다. 검증 실패 1회 재시도, 2회 연속 실패 시 에스컬레이션 시그널.
3. 측정값(해상도·여백·색상·가격)은 절대 LLM이 생성하지 않는다. LLM은 결정론적 리포트의 번역가다.
4. 상태 전이는 `state_machine.validate_transition`을 통과한 것만 커밋되고, 전부 이벤트 로그에 남는다.
